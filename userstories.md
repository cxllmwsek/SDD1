## 1. Product Backlog (User Stories)

**หลักการ:** อ้างอิงรูปแบบ As a... I want... So that... (Agile-2024.pdf หน้า 57-58) และบทบาท Product Owner ในการจัดการ Backlog (หน้า 13)

| ID | User Story | ผู้รับผิดชอบเขียน<br>(Role: Product Owner) |
| :--- | :--- | :--- |
| **US-01** | **As a** พนักงานหน้าร้าน (Staff)<br>**I want** บันทึกรายการออเดอร์ของลูกค้า<br>**So that** ส่งข้อมูลให้บาริสต้าทำเครื่องดื่มและคิดเงินได้ถูกต้อง | Product Owner |
| **US-02** | **As a** ลูกค้า (Customer)<br>**I want** สั่งออเดอร์ผ่านระบบออนไลน์และชำระเงินก่อน<br>**So that** ความสะดวกรวดเร็วและไม่ต้องรอคิวหน้าร้าน | Product Owner |
| **US-03** | **As a** เจ้าของร้าน (Owner)<br>**I want** จัดการข้อมูลเมนู ราคา และโปรโมชั่น<br>**So that** ให้ข้อมูลในระบบเป็นปัจจุบันและดึงดูดลูกค้า | Product Owner |
| **US-04** | **As a** พนักงานหน้าร้าน (Staff)<br>**I want** ให้ระบบตัดสต็อกวัตถุดิบทันทีที่ออเดอร์สำเร็จ<br>**So that** ป้องกันปัญหาสินค้าหมดโดยไม่รู้ตัว | Product Owner |
| **US-05** | **As a** ลูกค้าสมาชิก (Member)<br>**I want** สะสมแต้มจากยอดชำระเงิน<br>**So that** ใช้แลกสิทธิพิเศษในอนาคต | Product Owner |
| **US-06** | **As a** เจ้าของร้าน (Owner)<br>**I want** ดูรายงานยอดขายและวิเคราะห์ข้อมูล<br>**So that** วางแผนธุรกิจต่อไป | Product Owner |

## 2. การเรียงลำดับความสำคัญ (Prioritization)

**หลักการ:** เรียงตาม Value และความจำเป็นพื้นฐานเพื่อให้ได้ Working Software (Agile-2024.pdf หน้า 4, 6)

1.  **US-01 (บันทึกออเดอร์หน้าร้าน):** **สำคัญที่สุด (High Priority)**
    * เหตุผล: เป็นฟังก์ชันหลักของธุรกิจ (Core Business) หากขายของไม่ได้ ระบบอื่นก็ไม่มีประโยชน์ (Customer Satisfaction - หน้า 6)
2.  **US-03 (จัดการเมนู/ราคา):** **สำคัญรองลงมา (High Priority)**
    * เหตุผล: เป็นข้อมูลตั้งต้น (Master Data) ที่จำเป็นต้องมีก่อน จึงจะสามารถบันทึกออเดอร์ใน US-01 ได้
3.  **US-04 (ตัดสต็อก):** **สำคัญปานกลาง (Medium Priority)**
    * เหตุผล: เป็นกฎทางธุรกิจ (Business Rule) ที่สำคัญเพื่อความถูกต้องของการบริหารจัดการ แต่ไม่ใช่หน้าด่านในการขาย
4.  **US-02 (สั่งออนไลน์):** **สำคัญปานกลาง (Medium Priority)**
    * เหตุผล: เป็นฟีเจอร์ขยายช่องทางจำหน่าย ควรทำหลังจากระบบหน้าร้านเสถียรแล้ว (Sequential Integration - หน้า 75)
5.  **US-05 (ระบบสมาชิก):** **ความสำคัญต่ำ (Low Priority)**
    * เหตุผล: เป็นฟีเจอร์เสริม (Value Add) สามารถรอได้
6.  **US-06 (รายงาน):** **ความสำคัญต่ำสุด (Lowest Priority)**
    * เหตุผล: ต้องมีข้อมูล Transaction จากข้ออื่น ๆ ก่อนจึงจะสร้างรายงานได้

## 3. การเลือก User Stories เข้าสู่ Sprint 1

**เกณฑ์การเลือก:** เลือกงานที่ทีมทำไหวใน 1 Sprint และต้องส่งมอบเป็น Working Software ได้ (Agile-2024.pdf หน้า 4)

**User Stories ที่เลือก:**
1.  **US-03 (จัดการเมนู/ราคา):** เพื่อสร้างฐานข้อมูลสินค้า
2.  **US-01 (บันทึกออเดอร์หน้าร้าน):** เพื่อให้ร้านเริ่มขายของได้จริง (MVP)

## 4. การวางแผน Sprint (Sprint Planning)

### 4.1 Sprint Backlog
รายการที่ทีมจะทำใน Sprint นี้:
1.  **US-03:** จัดการข้อมูลเมนู ราคา และโปรโมชั่น
2.  **US-01:** บันทึกรายการออเดอร์ของลูกค้า

### 4.2 Sprint Tasks & 4.3 ผู้รับผิดชอบ
**หลักการ:** แตกงานเป็น Task ย่อยพร้อมประเมินชั่วโมง (หน้า 63) และใช้เทคนิค XP เช่น Unit Test, Simple Design (หน้า 71, 76)

#### **A. สำหรับ User Story: US-03 (จัดการเมนู)**

| Task Description | Est. Hours | ผู้รับผิดชอบ (Role: Developer) |
| :--- | :---: | :--- |
| 1. ออกแบบ Database สำหรับเก็บเมนูและราคา | 4 | Developer |
| 2. ออกแบบหน้าจอ (UI) สำหรับเพิ่ม/แก้ไข เมนู (Simple Design) | 4 | Developer |
| 3. เขียน Unit Test สำหรับฟังก์ชันเพิ่มเมนู (TDD - Test First) | 4 | Developer |
| 4. เขียน Code ฟังก์ชันเพิ่ม/แก้ไข/ลบ เมนู | 8 | Developer |
| 5. Integrate Code เข้า Repository | 2 | Developer |

#### **B. สำหรับ User Story: US-01 (บันทึกออเดอร์)**

| Task Description | Est. Hours | ผู้รับผิดชอบ (Role: Developer) |
| :--- | :---: | :--- |
| 1. ออกแบบหน้าจอรับออเดอร์ (UI) | 4 | Developer |
| 2. เขียน Unit Test การคำนวณราคาสินค้า (Test First) | 6 | Developer |
| 3. เขียน Code บันทึกออเดอร์ลงฐานข้อมูล | 8 | Developer |
| 4. เขียน Code เชื่อมโยงข้อมูลเมนูมาแสดงหน้าสั่งซื้อ | 6 | Developer |
| 5. ทดสอบระบบรวม (System Test/Acceptance Test) | 4 | Developer |
